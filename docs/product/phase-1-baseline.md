# 阶段一产品基线

## 阶段目标

阶段一目标是建设一个可运行、可验证、可扩展的个人股票研究工作台雏形。

当前最重要的不是堆满数据库后台功能，而是先把数据可信闭环做好，再把股票查看体验整理成每天愿意打开的研究入口。数据库、同步、批次、质量和血缘是后台能力，服务于股票、行情和后续新闻研究。

## 阶段一能力收敛

| 能力 | 说明 |
| --- | --- |
| 数据可见 | 用户能看清股票池、单股日线、数据源、任务、批次、质量和数据库状态 |
| 数据可同步 | 用户能创建股票池、日线、交易日历、市场补齐等同步任务 |
| 数据可追溯 | 正式数据能回到任务、批次、真实 provider、请求范围、schema 和质量结果 |
| 数据可治理 | 系统能展示覆盖率、缺失、重复、异常、容量、数据目录和水位线 |
| 股票可研究 | 用户能从股票池进入单股详情，看到基础信息、行情数据、覆盖、质量和批次 |

## 两层功能范围

| 层级 | 功能区 | 阶段一范围 |
| --- | --- | --- |
| 股票研究台 | 总控台 | 系统健康摘要、数据源健康、最近任务、质量告警、快捷入口 |
| 股票研究台 | 股票池 | A 股列表、搜索筛选、服务端分页、股票详情、日线表、覆盖和质量信息 |
| 股票研究台 | 新闻汇总 | 作为研究入口保留，当前不假装已有真实新闻主链路，下一步补新闻数据闭环 |
| 股票研究台 | 数值数据 | 作为行情和指标入口保留，当前不扩展成完整因子平台 |
| 数据可信后台 | 数据源管理 | 五类 provider 的展示、启用禁用、优先级、健康检查、真实取样 |
| 数据可信后台 | 同步调度 | 同步任务列表、定时配置、日志、批次、单股日线、市场补齐 |
| 数据可信后台 | 数据库管理 | 存储状态、数据集目录、血缘、质量报告、水位线、覆盖缺口和修复入口 |

## V1 验收口径

V1 完成时应满足：

- 可以查看、搜索和分页浏览 A 股股票池。
- 可以打开单只股票详情，看到基础信息、日线数据、覆盖情况、质量和入库批次。
- 可以管理 AKShare、BaoStock、AData、Tushare、Stock SDK。
- 可以查看 provider metadata、capabilities、health status 和真实取样。
- 可以创建股票池同步、单股日线同步、交易日历同步和市场级日线缺口补齐任务。
- 日线缺口以最近半年为默认管理窗口，并能从数据库管理页跳转到市场补齐任务。
- 可以查看同步任务状态、日志、入库批次、真实来源、写入量和失败原因。
- 可以在数据库管理里看到元数据库、数据湖、数据集、质量、血缘、水位线和覆盖缺口。
- 后续研究、因子、回测、策略模块不需要重新接第三方数据源。

## P1 产品增强验收口径

完成 V1 后，P1 重点从“能管理数据”转向“能研究股票”：

- 总控台更像研究首页，而不是后台入口集合。
- 股票详情页聚合价格、成交量、覆盖、质量、批次和后续新闻。
- 新闻汇总接入真实新闻数据，支持按股票、来源和日期过滤。
- 数据库管理页保留后台属性，只展示数据可信度、缺口、质量和修复入口。
- 页面文案减少抽象介绍，直接说明当前状态和下一步操作。

## 边界约束

- 前端只依赖 API，不直接访问数据库、数据湖路径或数据源 adapter。
- API 只创建长任务，不在请求线程里执行长时间同步。
- 正式写入必须经过 normalize、schema 校验、批次记录和质量检查。
- 单股日线同步必须有股票代码。
- 全市场缺口补齐必须使用 `daily_bars_market_repair`，不能用空 `symbol` 代表全市场。
- 数据源配置只属于数据系统，不混入通用系统设置。
- 新闻能力进入正式范围时，必须有标准化、批次、质量和股票关联，不做不可追溯的临时爬取。
