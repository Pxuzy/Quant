# 个人股票研究工作台产品方向

## 产品定位

Quant 的主目标不是做一个大型数据库后台，而是做一个给个人使用的股票数据管理与研究工作台。

数据库、数据源、同步任务、质量报告和血缘是控制面。用户每天真正需要打开的是工作台入口：看关注股票、看价格和成交量、看新闻事件、确认本地数据是否新鲜可靠。

一句话定位：

```text
Quant 是一个本地优先的个人股票数据管理与研究工作台，用可信的数据管理能力支撑股票、行情、新闻和数值分析。
```

## 两层产品结构

系统分成两层。以后新增功能先判断属于哪一层，避免把所有东西堆在数据库管理页里。

| 层级 | 面向问题 | 页面和能力 |
| --- | --- | --- |
| 工作台层 | 今天看什么股票，为什么要看，数据和新闻说明了什么 | 总控台、股票池、股票详情、数值数据、新闻汇总、后续自选股 |
| 控制面 | 数据从哪里来，是否新鲜，缺什么，怎么修 | 数据源管理、同步调度、数据库管理、数据质量、批次和血缘 |

工作台层是默认入口。控制面是维护入口，只在数据缺失、过期、失败或需要追溯时进入。

## 核心用户流程

阶段一到阶段二的体验围绕三个闭环建设。

### 1. 数据可信闭环

用户打开系统后，可以看到：

- 本地股票池数量。
- 日线数据最新交易日。
- 最近半年日线覆盖率和缺口数量。
- provider 健康状态。
- 最近同步任务和失败原因。
- 一键跳转到补齐任务。

这个闭环回答“我现在看到的数据能不能信”。

### 2. 股票查看闭环

用户从总控台或股票池进入单只股票，能看到：

- 股票基础信息。
- 日线价格和成交量。
- 最近同步批次。
- 数据覆盖和质量状态。
- 后续接入新闻后的股票相关新闻和事件时间线。

这个闭环回答“这只股票最近发生了什么，数据是否完整”。

### 3. 新闻事件闭环

新闻不是装饰页，而是工作台的一部分。后续新闻能力按下面的最小闭环建设：

- 拉取新闻文章。
- 标准化标题、来源、发布时间、正文摘要和链接。
- 关联股票代码。
- 在总控台、新闻汇总和股票详情页展示。
- 在数据库管理页展示新闻数据集的新鲜度、行数、批次和质量。

这个闭环回答“哪些新闻可能影响我关注的股票”。

## 页面方向

| 页面 | 方向 |
| --- | --- |
| 总控台 | 研究入口和系统健康摘要，不做营销式大屏 |
| 股票池 | 快速筛选和进入股票详情，后续支持自选股 |
| 股票详情 | 股票研究主页面，聚合基础信息、行情、新闻、质量和批次 |
| 新闻汇总 | 从占位页演进为新闻事件流，支持按股票、来源、日期过滤 |
| 数值数据 | 聚合常用行情和指标，不提前扩展成完整因子平台 |
| 数据源管理 | 管 provider 能力、健康、取样和优先级 |
| 同步调度 | 管任务、日志、批次、worker 和补齐计划 |
| 数据库管理 | 管数据集、覆盖、质量、水位线、血缘和修复入口 |

## 界面原则

界面应该像研究和运维工具，不像宣传页。

- 信息密度适中，方便扫描和比较。
- 少用大卡片堆叠，优先使用状态条、表格、图表、时间线和紧凑摘要。
- 颜色只表达状态：正常、警告、失败、运行中。
- 每个页面只保留一到两个主操作。
- 页面文案直接说明当前状态和下一步，不写空泛介绍。
- 数据管理页面要突出“哪里缺、怎么补、补到哪里”，不要展示无关技术细节。

## 近期优先级

| 优先级 | 目标 | 说明 |
| --- | --- | --- |
| P0 | 稳定数据可信闭环 | provider、日线、交易日历、批次、质量、最近半年补齐 |
| P1 | 重塑股票研究入口 | 总控台和股票详情从“数据系统入口”改成“股票研究入口” |
| P1 | 接入新闻最小闭环 | 新闻文章、股票关联、新闻汇总、股票详情新闻区 |
| P1 | 优化页面信息架构 | 减少重复请求和重复卡片，突出状态、缺口和下一步 |
| P2 | 自选股和关注列表 | 让用户每天从自己的股票列表开始 |
| P2 | 指标和因子扩展 | 在行情数据稳定后再做，不抢当前阶段 |

## 暂不做

- 不做大型数据湖平台。
- 不做插件市场。
- 不做复杂权限和多租户。
- 不做完整回测引擎。
- 不做实盘交易。
- 不做 tick、orderbook、实时流。
- 不把新闻系统做成通用爬虫平台。

## 判断标准

一个新增功能只有满足下面至少一项，才应该进入近期范围：

- 能帮助用户更快判断某只股票是否值得看。
- 能让行情或新闻数据更可信。
- 能减少补数、追溯和排错成本。
- 能让页面更清晰、更快、更少误导。

否则先放进候选池，不进入当前实现。
