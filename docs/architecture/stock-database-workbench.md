# 股票数据库工作台项目文档

日期：2026-06-07

## 第一版项目目标简洁版

本节是当前第一版项目文档的最高层目标。后续开发优先阅读本节，下面的详细架构用于展开实现细节；如果两者出现表述冲突，优先按本节的产品目标收敛。

### 1. 一句话目标

先做一个桌面端 Web 版 **股票数据库工作台**：能管理 A 股股票池、同步真实行情数据、查看单股数据、管理数据源、监控数据库状态，为后续因子、回测、策略系统提供干净的数据底座。

### 2. 核心数据链路

```text
AKShare / BaoStock / AData / Tushare / Stock SDK
  -> 数据源适配
  -> 标准化
  -> 校验
  -> 入库批次
  -> PostgreSQL 元数据
  -> Parquet 行情数据
  -> DuckDB 查询
  -> 前端展示
```

关键原则：

- 前端不直接调用 AKShare、BaoStock、AData、Tushare、Stock SDK。
- 后续因子、回测、策略也不直接调用数据源。
- 所有正式数据都必须能追溯：来源、任务、批次、日期范围、写入数量、质量状态。

### 3. 五个主功能

#### 3.1 总控台

看系统整体状态。

需要显示：

- 股票池数量
- 日线最新日期
- 数据源健康状态
- 最近同步任务
- 数据库容量
- 数据质量异常

需要提供快捷入口：

- 更新股票池
- 更新日线
- 查看数据源
- 查看数据库状态

#### 3.2 股票池

管理全部 A 股股票。

列表字段：

- 代码
- 名称
- 交易所
- 行业
- 上市状态
- 最新数据日
- 数据完整度

点击股票进入详情页。

股票详情页包含：

- 基础信息
- 日线图
- 日线数据表
- MA5 / MA10 / MA20
- 涨跌幅
- 数据质量
- 新闻占位

#### 3.3 数据源管理

只管理第一版明确需要的数据源：

- AKShare
- BaoStock
- AData
- Tushare
- Stock SDK

需要支持：

- 启用 / 禁用
- 优先级
- 健康检查
- 真实取样
- Token / 可选包状态
- 支持能力展示

不做插件市场，不展示测试数据源。

#### 3.4 同步调度

负责“什么时候从哪里抓什么数据”。

第一版支持：

- 同步股票池
- 同步单只股票日线
- 同步指定日期范围日线
- 市场级日线缺口补齐
- 查看任务记录
- 查看失败原因和日志

注意：

- 单股日线同步必须填写股票代码。
- 全市场补齐必须是单独任务，不能用空股票代码代替。

#### 3.5 数据库管理

查看本地数据资产状态。

需要显示：

- 股票基础表数量
- 日线数据数量
- 日线最新日期
- 交易日历覆盖范围
- 数据质量报告
- 数据目录
- 同步水位线
- PostgreSQL / SQLite 状态
- Parquet 容量
- DuckDB 查询状态

数据集目录、交易日历、数据质量报告都放到这里，不单独做主导航。

## 0. 项目目标基线

当前项目目标是建设一个桌面端 Web 版“股票数据库工作台”。它是后续因子研究、策略、回测和交易模块的数据底座，不是完整量化平台，也不是一组分散的后台页面。

> 目标状态：本章已确认作为第一阶段总架构目标。后续页面、API、数据源接入、同步任务、数据库管理和质量治理，都优先按本章验收。

后续实现、页面调整、数据源接入、数据库设计和质量治理，都默认以本文件为目标基线。任何新增功能都要先判断是否服务下面四件事：

- 数据可见：用户能看清股票池、单股日线、指标、来源、质量和数据库状态。
- 数据可同步：用户能手动或定时从 AKShare、BaoStock、AData、Tushare、Stock SDK 等来源抓取股票池、交易日历和日线数据。
- 数据可追溯：每条正式写入的数据都能回到同步任务、入库批次、真实 provider、请求范围、schema 版本和质量结果。
- 数据可治理：系统能展示最新日期、覆盖率、缺失、重复、异常价格、容量、数据目录和同步水位线。

第一阶段的目标架构固定为五个主入口：

```text
总控台
股票池
数据源管理
同步调度
数据库管理
```

如果新需求不能自然落到这五个入口之一，优先放到后续“研究、回测、交易”阶段，不进入第一阶段主导航。

### 0.1 项目总架构目标（第一阶段基准）

第一阶段的总架构目标是把项目做成一个“股票数据库工作台”。下面这套架构不是临时建议，而是当前开发、验收和后续扩展的基准：前端是桌面 Web 数据管理产品，后端是模块化数据底座，数据层能支撑后续因子、研究、回测、策略和交易模块。

```text
桌面 Web 工作台
  总控台 / 股票池 / 数据源管理 / 同步调度 / 数据库管理
    -> 前端应用层
       React + Vite + TypeScript + Ant Design + TanStack Query + TanStack Router + ECharts
    -> API 契约层
       FastAPI 只暴露业务 API，不泄露 provider 字段、数据库表结构和 Parquet 路径
    -> 领域服务层
       stocks / market_data / data_sources / sync_tasks / datasets / data_quality / calendars
    -> 数据接入层
       provider registry + capabilities + adapter.fetch + normalize + schema_validate + fallback
    -> 数据治理层
       ingest_batches + task_logs + lineage + quality_reports + catalog + watermarks
    -> 存储查询层
       PostgreSQL 元数据 / SQLite 本地 fallback / Parquet 行情数据湖 / DuckDB 查询引擎
    -> 后续量化模块
       因子 / 研究 / 回测 / 策略 / 交易，只消费统一数据服务
```

这张总架构图就是第一阶段开发目标。页面、API、数据源接入、同步任务和数据库管理都要围绕它收敛：

- 产品入口只保留五个，避免把后端模块平铺成菜单。
- 前端只编排业务体验，不直接理解第三方数据源、数据库表和 Parquet 文件。
- API 只暴露稳定业务契约，长时间任务交给 worker 或轻量任务执行器。
- 数据源只是入口，正式数据必须经过标准化、字段校验、同步批次、质量检查和目录登记。
- PostgreSQL 是生产元数据库目标；SQLite 是本地开发 fallback；Parquet 保存行情大表；DuckDB 负责查询 Parquet；SQL 是查询语言，不是另一个独立数据库。
- 后续因子、研究、回测、策略和交易模块不能直接调用第三方数据源，也不能直接拼数据湖路径，只能消费统一 API、silver/gold 数据集或稳定读取接口。

每次新增功能都按下面问题验收：

1. 是否落在五个主入口之一。
2. 是否让数据更可见、可同步、可追溯或可治理。
3. 是否保持了前端、API、领域服务、数据接入和存储查询的边界。
4. 是否避免把第一阶段做成完整量化平台、通用数据平台或运维后台。

## 1. 项目定位

本项目第一阶段要做的是一个“股票数据库工作台”，不是一堆平铺的后台菜单。它的核心目标是把 A 股股票基础数据、日线行情、数据源、同步调度、数据质量和存储状态组织成一个可使用、可扩展、职责清晰的数据管理产品。

第一版只做股票数据底座，不做移动端、不做完整回测、不做实盘交易，也不把 Redis、MongoDB、Celery 等基础设施作为必需项。后续因子、策略、回测和交易模块都应依赖这个数据工作台提供的统一数据服务。

## 2. 目标展开说明

本节展开说明 `0.1 项目总架构目标`，不重新定义另一套目标。后续实现、页面重构、数据源接入和数据库治理功能都要优先对齐第 0 章；如果后续需求与第 0 章冲突，需要重新确认后再调整。

### 2.1 产品形态

系统目标是一个桌面端 Web 可视化数据管理工作台，而不是一组平铺的后台菜单，也不是完整量化研究或交易平台。

它要同时服务两类高频问题：

- 股民视角：我有哪些股票，某只股票的基础资料、日线、指标和数据质量怎么样。
- 数据管理视角：数据从哪里来，什么时候同步，写到了哪里，现在是否完整、可信、可追溯。

第一阶段只围绕 A 股股票池、日线行情、数据源、同步批次、数据质量和数据库状态形成闭环。

### 2.2 前端总架构

主导航只保留五个入口：

```text
总控台
股票池
数据源管理
同步调度
数据库管理
```

五个入口的产品分工：

- 总控台：运行态总览和快捷操作，只展示关键状态，不承载深层配置。
- 股票池：A 股股票列表和单股详情，是看股票和钻取行情数据的主入口。
- 数据源管理：管理 AKShare、BaoStock、AData、Tushare、Stock SDK 的状态、优先级、能力、凭证/可选包状态和真实取样。
- 同步调度：表达“什么时候从哪里抓什么数据”，展示手动同步、定时配置、任务记录、日志和同步批次。
- 数据库管理：展示元数据库、Parquet 数据湖、DuckDB 查询能力、交易日历、数据目录、质量报告、同步水位线和数据覆盖率。

隐藏到五个入口内部的能力：

- 日线行情：进入股票详情页和同步调度，不作为主导航。
- 数据集目录：进入数据库管理。
- 交易日历：进入数据库管理和同步调度。
- 数据质量：进入数据库管理和股票详情。

### 2.3 数据总架构

数据链路采用“外部来源 -> 标准化 -> 入库批次 -> 本地标准数据层 -> 质量治理 -> API 展示”的结构：

```text
AKShare / BaoStock / AData / Tushare / Stock SDK
  -> provider adapter 拉取
  -> normalize 标准化
  -> schema_validate 字段校验
  -> ingest_batch 形成同步批次
  -> PostgreSQL / SQLite 写元数据
  -> Parquet 写行情大表
  -> DuckDB 查询 Parquet
  -> quality_check 质量检查
  -> catalog / lineage 更新目录和血缘
  -> API 给前端、研究、因子和回测使用
```

关键原则：

- 数据源只是入口，本地标准数据层才是系统核心。
- `auto` 是来源选择模式，不是实际 provider；正式写入必须记录真实 provider。
- 第三方接口返回的数据不能直接给前端或后续策略使用，必须经过标准化、校验、批次记录和质量检查。
- 每次正式同步都要能追溯到数据源、请求范围、写入数量、schema 版本、normalize 版本、质量状态和错误原因。
- 后续因子、回测和策略模块只能依赖 API、silver/gold 数据集或稳定读取接口，不能直接调用第三方数据源，也不能直接拼 Parquet 路径。

### 2.4 存储和查询边界

第一阶段采用元数据和行情大表分离：

- PostgreSQL：生产目标元数据库，保存股票基础资料、数据源配置、同步任务、入库批次、数据目录、质量报告和用户可见配置。
- SQLite：本地开发和 bootstrap fallback，只作为 PostgreSQL 未就绪时的轻量替代。
- Parquet：行情大表和后续因子矩阵的数据湖存储，适合日线、分钟线、复权行情和批量分析。
- DuckDB：查询 Parquet 的执行引擎，用于本地快速预览、聚合、质量扫描和研究型查询，不作为持久化主数据库。
- SQL：查询语言或接口能力，不代表另一个独立数据库。

### 2.5 开源项目借鉴目标

本项目只吸收成熟量化项目在数据工程上的稳定做法，不照搬它们的完整平台：

- Qlib：先形成本地标准数据层，再给研究、指标、模型和回测使用。
- Zipline：把同步视为 ingest/bundle，保留可重复执行的批次、快照和失败记录。
- OpenBB：统一 provider metadata、参数模型、返回结构和字段校验。
- vn.py：用统一 datafeed/provider 接口隔离具体数据源，让替换 provider 不影响上层功能。
- LEAN：重视资产身份、复权口径和公司行动，第一阶段至少显式记录 `adjust_type`，后续预留公司行动模型。
- backtrader：让策略或回测消费稳定 data feed，而不是消费第三方下载函数。

第一阶段不做完整 Qlib 因子表达式引擎、Zipline bundle 管理、OpenBB 插件市场、LEAN 实时订阅、vn.py 交易网关或 backtrader 回测运行时。

## 4. 功能设计

### 4.1 总控台

总控台是统一入口，不只是看板。

需要显示：

- 股票池数量
- 日线最新日期
- 数据源健康状态
- 同步任务状态
- 数据库容量
- 异常提醒

需要提供快捷操作：

- 更新股票池
- 更新日线
- 查看数据源
- 查看数据库状态

### 4.2 股票池

股票池存放全部 A 股股票。

列表字段：

- 代码
- 名称
- 交易所
- 行业
- 上市状态
- 最新数据日
- 数据完整度

点击单只股票进入股票详情页。

股票详情页第一版包含：

- 基础信息：代码、名称、交易所、行业、上市日期、状态。
- 日线图：K 线、收盘趋势、成交量。
- 日线数据表：OHLCV、成交额、来源、更新时间。
- 指标：MA5、MA10、MA20、涨跌幅、成交量均值。
- 数据质量：缺失交易日、重复记录、异常价格、复权口径、数据来源批次。
- 新闻：后续能力先留占位，不接真实新闻数据，第一阶段不把详情页扩成资讯页。

后续指标再扩展 MACD、RSI 等，不在第一版强做。

### 4.3 数据源管理

第一版只管理明确采用的数据源：

- AKShare
- BaoStock
- Tushare

显示信息：

- 启用状态
- 优先级
- 支持能力
- 是否需要 Token
- 健康状态
- 最近取样结果

支持操作：

- 健康检查
- 真实取样
- 调整优先级
- 启用或禁用

数据源管理页要借鉴 Qlib、Zipline、vn.py、OpenBB 这类项目的做法，但只展示第一版需要的产品能力：

- 每个数据源都必须声明支持能力，例如股票池、日线行情、交易日历。
- 用户可以看到数据源是免凭证还是需要 Token，以及当前是否可用。
- “自动选择”不是一个真实 provider，而是按启用状态、健康状态、能力和优先级选择具体数据源。
- 真实取样只拉小样本，用来确认接口、字段映射和标准化结果，不直接当作正式入库数据。
- 每次正式同步要记录实际来源、同步范围、写入数量、失败原因和数据质量结果。
- 数据源只是入口，正式业务只使用系统标准 schema、同步批次、数据目录和质量报告，不直接依赖第三方接口。
- 多个数据源给出同一数据但结果不一致时，系统要保留实际来源和差异记录，不能静默拼成一个不可追溯的数据集。

不做复杂插件市场，不暴露未纳入第一版的数据源、演示型 provider 或测试型 provider。

### 4.4 同步调度

同步调度表达的是“什么时候从哪里抓什么数据”，不是单纯任务列表。

第一版需要支持：

- 手动同步股票池。
- 手动同步单只股票日线。
- 手动同步单只股票的指定日期范围日线。
- 市场级日线缺口补齐：从数据库管理的水位线和覆盖率缺口进入，按市场、日期范围和安全阈值创建受控补齐任务。
- 定时同步配置占位：每天收盘后更新日线、每周更新股票池。
- 最近任务记录：成功/失败、写入数量、失败原因、日志。

当前 `sync_tasks` 后端模块可以保留，前端产品命名统一为“同步调度”或“同步记录”。

日线同步必须区分两种语义：

- 单股日线同步：任务类型为 `daily_bars`，必须填写股票代码，适合股票详情页或单只股票补数。
- 市场级日线缺口补齐：目标任务类型为 `daily_bars_market_repair`，不接收股票代码，由后端根据股票池、交易日历和已有日线数据计算缺口，再受控地逐只股票补齐。

不能把 `daily_bars` 的空 `symbol` 解释成全市场同步；这种写法会让任务语义、日志、批次和质量追溯变得不清楚。

同步调度页要表达完整数据接入链路：

```text
选择数据类型和范围
  -> 选择自动来源或指定来源
  -> 创建同步任务
  -> worker 拉取并标准化
  -> 写入数据库/数据湖
  -> 更新质量和目录状态
```

页面不需要暴露 adapter 内部细节，但要能让用户知道：本次从哪个数据源写入、有没有降级到备用来源、失败在哪一步、是否写入了主数据。

同步调度需要逐步补齐标准数据整合信息：

- 同步批次：`ingest_batch_id`、实际数据源、请求数据源、数据类型、范围、写入数量。
- 数据契约：schema 版本、normalize 版本、字段校验结果。
- 同步水位线：每类数据、每个市场或股票的最近成功同步日期、最近失败原因和可修复范围。
- 多源降级记录：自动选择时尝试了哪些来源、哪个来源最终写入、失败来源的原因。
- 补齐计划：市场级补齐需要显示计划股票数、计划交易日数、预计缺口数、安全阈值、实际写入批次和失败位置。

### 4.5 数据库管理

数据库管理展示数据库和数据湖状态。

需要显示：

- 股票基础表有多少条。
- 日线数据有多少条。
- 日线最新日期到哪天。
- 交易日历覆盖到哪天。
- 数据质量状态。
- 数据集主存储介质：元数据、目录和任务状态类数据集使用 PostgreSQL；行情大表使用 Parquet；DuckDB 是查询 Parquet 的执行引擎，不作为持久化主存储。
- 容量：PostgreSQL 元数据库容量；本地开发 fallback 时显示 SQLite 文件大小；行情数据湖显示 Parquet 文件大小。
- 数据版本和来源：最近一次同步批次、实际 provider、schema 版本、质量检查结果。
- 数据新鲜度：股票池、交易日历、日线行情分别更新到哪一天。
- 数据覆盖率：A 股股票池覆盖、交易日历覆盖、日线缺失情况。
- 数据版本/快照：可用于研究或回测的数据集版本、分区清单和最近变更。
- 多源差异：不同数据源在同一股票、同一交易日上的差异和处理状态。
- 同步水位线：各类数据最近成功同步位置和待补数据范围。
- 市场级补齐入口：当数据库管理发现“日线只覆盖部分股票”或“某日期范围存在大量缺口”时，跳转到同步调度创建 `daily_bars_market_repair`，而不是创建缺少股票代码的单股同步任务。

合并到数据库管理里的二级功能：

- 数据集目录
- 交易日历
- 数据质量报告

## 5. 后端模块边界

后端可以继续保留清晰的领域模块，前端不需要按后端模块一比一铺菜单。

- `stocks`：股票池和股票详情基础信息。
- `market_data`：日线行情查询、日线同步。
- `data_sources`：数据源配置、健康检查、真实取样。
- `sync_tasks`：同步任务、定时计划、日志。
- `datasets/catalog`：数据库管理里的数据资产清单。
- `data_quality`：数据库管理和股票详情里的质量状态。
- `news`：预留模块，第一版只做占位。

数据整合层必须保持标准化边界：

- `data_sources` 只负责 provider registry、能力声明、健康检查、真实取样和标准化输出。
- `sync_tasks` 负责任务生命周期、调度记录、重试、日志和同步水位线；日线单股同步和市场级缺口补齐必须使用不同任务语义。
- `ingest_batches` 负责记录每次正式写入的批次、来源、schema/normalize 版本、写入数量和错误。
- `datasets/catalog` 负责记录数据集、版本、schema、分区、血缘、最新日期和质量状态。
- `market_data` 对后续指标、因子、回测暴露稳定读取接口，后续可演进成 BarReader/DataPortal，不让上层直接读 Parquet 路径或第三方接口。

## 6. 第一版闭环

第一版最重要的是先把真实股票数据库闭环跑通：

1. 主导航重构成五个入口。
2. “股票库”统一改成“股票池”。
3. 增加单股详情页。
4. 单股详情页整合基础信息、日线图、日线表和简单指标。
5. 数据集目录、交易日历、数据质量合并到数据库管理。
6. 同步任务产品化为同步调度，后续再补定时规则。

数据层第一版同步采用的成熟项目经验：

1. 像 Qlib 一样，先形成本地标准数据层，再给研究、指标、回测使用。
2. 像 Zipline 一样，把同步过程视为 ingest/bundle，保留可回放批次和数据快照。
3. 像 LEAN 一样，提前重视资产身份、复权口径和公司行动，第一阶段先显式记录 `adjust_type`。
4. 像 OpenBB 一样，统一 provider metadata、参数、返回结构和字段校验。
5. 像 vn.py 一样，用统一 datafeed/provider 接口隔离具体数据源。
6. 像 backtrader 一样，让后续策略或回测消费稳定 data feed，而不是消费下载函数。

第一版不照搬完整量化平台能力：不做完整因子表达式引擎、不做完整回测运行时、不做实时行情订阅、不做插件市场、不做 tick/orderbook 实时流。先把 A 股股票池、交易日历、日线、质量、目录和同步批次跑稳。

## 7. 工程约束

- 使用 React + Vite + TypeScript + Ant Design + ProComponents 建设桌面管理后台。
- 前端只通过 API 获取数据，不直接理解数据源适配器、数据库表或 Parquet 路径。
- 大表必须服务端分页或受限预览，不能一次性拉取全量行情。
- 数据源配置只属于数据系统，不混入通用系统设置。
- API 只创建任务，不执行长时间同步任务；长任务由 worker 执行。
- 页面以中文为主，保留 AKShare、BaoStock、AData、Tushare、Stock SDK、PostgreSQL、Parquet、OHLCV、MA5 等必要技术名。
- 每个功能必须有 loading、empty、error 状态。
- 每次前端功能完成后运行类型检查、构建，并用浏览器验证页面可用。
